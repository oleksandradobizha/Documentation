Playwright Automation Framework

End-to-end test automation for the Buildee web application — a building energy management platform. Built with Playwright and TypeScript, following the Page Object Model.

Overview

The framework is modular with reusable components for pages, utilities, and test cases. It covers login flows, building management, equipment CRUD, utility data entry/import, measures, proposals, projects, reports, and task lists — and integrates with TestRail and Slack.

Getting Started

Prerequisites

  • Node.js (LTS, v16+)
  • npm (comes with Node.js)
  • Access to the Buildee application (staging or production URL)
  • Valid Buildee user credentials

Installation

# Clone the repository
git clone https://github.com/simuwatt/automation-qa.git
cd automation-qa

# Install dependencies
npm install

# Install Playwright browsers
npx playwright install

Authentication Flow

  1. Pre-test setup — tests/auth.setup.ts authenticates once and saves the session to playwright/.auth/user.json.
  2. Session reuse — all chromium tests load the saved session, skipping login.
  3. Login tests — run in a separate project without stored state to exercise the real login flow.
  4. Re-authentication — reAuthenticateIfNeeded() in common.page.ts handles expired sessions automatically.

GitHub Actions Secrets

Configure these under Settings → Secrets and variables → Actions:

BASE_URL BASE_URL_PROD USERNAME USEREMAIL PASSWORD SLACK_WEBHOOK_URL USEREMAIL_TESTRAIL TRCLI_KEY

Running Tests

NPM Scripts

Predefined scripts use cross-env to inject test_env (which determines the loaded .env.[env] file):

# Run QA tests (default)
npm run env-qa

# Headed mode (browser visible)
npm run env-qa-headed

# Filter by tag (via grep)
npm run env-qa-tag

# Run a specific spec file
npm run env-qa addProposal.spec.ts

Playwright CLI Examples

# Run all tests
npx playwright test

# Run a specific spec
npx playwright test tests/e2e/Buildings/buildingsManagement.spec.ts

# Run by tag
npx playwright test --grep "@smoke_prod"

# Run everything except smoke production
npx playwright test --grep-invert "@smoke_prod"

# Headed + specific project
npx playwright test --headed --project=chromium

# Target a custom environment
test_env=staging npx playwright test

# Open the HTML report after a run
npx playwright show-report

VSCode Playwright Extension

You can also run or debug tests directly from the Playwright extension. It always uses the root .env file (no dynamic env switching).

Page Object Model

The framework follows a base-class hierarchy. All page objects extend BasePage, list/table views extend PortfolioTable.

BasePage

Root class with reusable UI interaction methods:

  • clickElementWithText(locator, text) — Click an element matching text
  • clickElementWithJS(locator) — Click using JavaScript (for hidden elements)
  • elementContainsText(locator, text, exact) — Assert text content
  • findElementByName(name) — Locate element by name attribute
  • toggleSwitchBox(locator) — Toggle a switch/checkbox
  • dragAndDropElements(source, target) — Drag-and-drop interaction

PortfolioTable

Shared class for interacting with list/table views (portfolios) across the app:

  • clickItem(name) — Click a row by item name
  • verifyItemIsPresent(name) — Assert an item exists in the table
  • verifyItemsArePresent(names[]) — Assert multiple items exist
  • verifyTotalCount(count) — Verify the displayed total
  • clickExtrasButton(name) — Click the extras/action button for an item
  • waitForTableToLoad() — Wait until the table finishes loading

TestRail Integration

The TestRail reporter (.github/scripts/report_testrail_results.js):

  • Creates test runs in TestRail
  • Maps results using case IDs embedded in test titles (e.g. C315, C1784)
  • Creates missing test cases on-the-fly (fallback — not ideal for traceability)
  • Attaches failure screenshots to test results
Currently disabled in CI (commented out). To re-enable, uncomment the relevant steps in staging.yml and configure the TestRail secrets.

Tags & Test Filtering

Tests use tags for selective execution. By default, everything without a specific tag is treated as regression. TestRail case IDs in test titles (e.g. C590 C591 C594) link tests to TestRail cases.

Extending the Framework

Add a New Page

  1. Create a file in pages/ (e.g. pages/NewPage.ts).
  2. Extend BasePage and define locators and methods.
  3. Extend PortfolioTable if the page uses a table view.

Add a New Test

  1. Create a spec in tests/e2e/ (e.g. tests/e2e/newTest.spec.ts).
  2. Import test from baseTest.ts and use the provided fixtures.
  3. Place shared data in testData.json; use test-specific JSON/CSV only when necessary.

Best Practices

  • Use Page Objects — encapsulate UI interactions in page classes.
  • Reuse components — use PanelFilter, PortfolioTable, and helpers for common interactions.
  • Tag tests — use tags (e.g. @smoke_prod) to group and filter.
  • Handle errors gracefully — guard flaky actions and empty states.
  • Never push to main — always branch and open a PR for review.
  • Verify stability before merging — run locally and, if needed, trigger GHA from your branch. You can temporarily edit the .yml file to run just the test(s) you need, and comment Slack notifications to avoid spam.
Note on session storage: reusing session storage is usually faster than logging in per test. However, in this web app, restoring the session is slower on GHA, so manual login is used by default. To re-enable, uncomment the setup dependency and storage lines in playwright.config.ts and remove the login step from the beforeEach hook.

Debugging Tips

  • Headed mode: run with --headed to watch the browser
  • Trace viewer: enable trace: "retain-on-failure" in playwright.config.ts, then npx playwright show-trace
  • Video replay: check test-results/ for recorded videos of failed tests
  • Pause on failure: add await page.pause() to open the Playwright Inspector
  • Single test: npx playwright test <file> -g "<test name>"

5. Test Case Management (TestRail)

5.1 General Rules

  • All new functionality must have test cases in TestRail.
  • Test cases should include:
    • Steps
    • Expected results
    • Clear structure

5.2 Automation Integration

  • Test runs are automatically created via GitHub Actions.
  • Automation is linked to TestRail test cases.

Automation Type Rules

Automation typeWhen to use
PlaywrightIf automated
NoneIf manual

5.3 Best Practice for Automation

When automating a new scenario:

  1. Create the test case in TestRail
  2. Copy the test case ID
  3. Add the ID to the automated test title
⚠️ Important Notes
  • If no test case exists, automation will create one automatically (fallback only).
  • This leads to duplicates and poor traceability.
❌ Do NOT:
  • Use fake IDs (e.g. C099999999)
  • Use non-existing IDs — test runs will fail
